## From the security re-review (`docs/SECURITY_REVIEW_l4-proxy-v0.1.0.md`)

### N1 — Low — Block IPv6 link-local and unique-local hosts in `validate_upstream_url`

- **Source:** Security re-review N1 (`docs/SECURITY_REVIEW_l4-proxy-v0.1.0.md:511-535`).
- **Where:** `crates/replayable-proxy/src/config.rs` — `validate_upstream_url` and `BANNED_UPSTREAM_HOSTS`.
- **Problem:** `https://[fe80::1]/` and `https://[fd00:ec2::254]/` (AWS IPv6 IMDS) are not blocked. The current deny-list is string-equality against four named hostnames and only handles the IPv4 link-local IMDS (`169.254.169.254`) by name.
- **Fix sketch** (from the review):
  ```rust
  use std::net::IpAddr;
  if let Some(url::Host::Ipv6(addr)) = parsed.host() {
      let seg0 = addr.segments()[0];
      let is_link_local = seg0 & 0xffc0 == 0xfe80;       // RFC 4291 fe80::/10
      let is_unique_local = seg0 & 0xfe00 == 0xfc00;     // RFC 4193 fc00::/7
      if (is_link_local || is_unique_local) && !IpAddr::V6(addr).is_loopback() {
          return Err(ConfigError::Invalid {
              name: ENV_UPSTREAM_URL,
              reason: format!("IPv6 link-local / unique-local host `{addr}` is not allowed"),
          });
      }
  }
  ```
- **Verification:** Add `h3_ipv6_link_local_rejected` and `h3_ipv6_unique_local_rejected` to `crates/replayable-proxy/tests/security.rs`.

### N3 — Informational — Defensive `fchmod(0o600)` on log file after open

- **Source:** Security re-review N3 (`docs/SECURITY_REVIEW_l4-proxy-v0.1.0.md:542-557`).
- **Where:** `crates/replayable-proxy/src/trace.rs` — `open_log_file` on Unix.
- **Problem:** `OpenOptions::mode(0o600)` only takes effect on the `O_CREAT` path. If a JSONL file already exists at `REPLAYABLE_LOG_PATH` with looser permissions (e.g. left over from a pre-fix run), the proxy will reopen it with its existing permissions.
- **Impact:** Upgrade-path edge only. Content capture has been off-by-default since v0.1.0, so any pre-existing file has no sensitive content to leak.
- **Fix sketch:** After open, call `fchmod` to force `0o600`:
  ```rust
  #[cfg(unix)]
  {
      use std::os::unix::fs::PermissionsExt;
      let mut perms = file.metadata().await?.permissions();
      perms.set_mode(0o600);
      file.set_permissions(perms).await?;
  }
  ```
- **Verification:** Add a test that pre-creates the log file with mode `0o644`, runs the proxy briefly, and asserts the file mode is `0o600` afterwards.

### N4 — Low — Extend `SCRUBBED_HEADER_NAMES` with provider-specific credentials

- **Source:** Security re-review N4 (`docs/SECURITY_REVIEW_l4-proxy-v0.1.0.md:559-570`).
- **Where:** `crates/replayable-proxy/src/proxy.rs` — `SCRUBBED_HEADER_NAMES` constant (lines 43-54).
- **Problem:** When `REPLAYABLE_CAPTURE_CONTENT=true`, the following credential-bearing headers are currently written verbatim to the JSONL trace because they are not in the scrub list:
  - `anthropic-api-key` — third-party Anthropic middleware / SDKs use this.
  - `x-goog-api-key` — Google Generative Language API key.
  - `x-amz-security-token` — AWS SigV4 session token (Bedrock).
- **Impact:** Bounded by C1's secure default (capture is off). Only matters when an operator opts in.
- **Fix sketch:** Append the three names to `SCRUBBED_HEADER_NAMES`. Update the README scrub-list documentation. Add `c1_capture_on_scrubs_<header>` assertions for each.
- **Skip:** `openai-organization`, `openai-project`, `x-goog-user-project`, `x-amz-date`, `x-amz-content-sha256` — not credential-bearing.

### M1 — Medium — Per-stream concurrency cap

- **Source:** Security review M1 (Medium, deferred from the original review per the "merge with caveats" guidance).
- **Where:** `crates/replayable-proxy/src/proxy.rs` request handler; no semaphore exists yet.
- **Problem:** A burst of concurrent requests can exhaust the host's open-file budget / reqwest connection pool. v0.1.0 has no upper bound on in-flight handler tasks.
- **Fix sketch:** Wrap the handler in a `tower::limit::ConcurrencyLimitLayer` (or a tokio `Semaphore` acquired before `body.collect()`), default to `512` concurrent in-flight requests, configurable via `REPLAYABLE_MAX_CONCURRENT_REQUESTS`. Requests above the cap should return `503 Service Unavailable` with `Retry-After: 1`.
- **Verification:** Load-test that asserts the proxy stays under N+1 open sockets at the upstream when N+M concurrent clients hit it.

### M2 — Medium — Per-stream aggregate body cap on SSE responses

- **Source:** Security review M2 (Medium, deferred).
- **Where:** `crates/replayable-proxy/src/proxy.rs` SSE pass-through path.
- **Problem:** The SSE pass-through aggregates the streamed response body into memory for the trace record. An adversarial or runaway upstream that streams gigabytes will exhaust process memory before the connection is closed.
- **Fix sketch:** Apply a configurable cap (`REPLAYABLE_MAX_RESPONSE_BYTES`, default 50 MiB) to the aggregated buffer. When exceeded, finish forwarding to the client but record `output_truncated: true` and a `bytes_dropped` count in the trace; do NOT terminate the client stream.
- **Verification:** Stream a 100 MiB SSE response through the proxy with a 50 MiB cap; client must see all 100 MiB; trace must show truncation.

### M4 — Medium — Audit-grade dropped-trace metric

- **Source:** Security review M4 (Medium, deferred).
- **Where:** `crates/replayable-proxy/src/trace.rs` — current drop counter is a `tracing::warn!` only.
- **Problem:** Dropped traces (when the mpsc channel is full) are logged as warnings only — no structured metric for alerting. Auditors cannot reconstruct how many requests were silently un-traced.
- **Fix sketch:** Expose a per-process counter (e.g. an `Arc<AtomicU64>`) and emit it as a structured field on every drop, then surface it through a future `/metrics` endpoint (Prometheus text format) or via a periodic INFO line.
- **Verification:** Drive the channel-full path; scrape the counter; confirm increments.

### L4 — Low — Base64 byte-exact fallback for non-UTF-8 bodies

- **Source:** Security review L4 (Low, deferred — forward-compat).
- **Where:** `crates/replayable-proxy/src/proxy.rs` — `String::from_utf8_lossy` on `model_calls[].input` / `.output`.
- **Problem:** When capture is on and the body is not valid UTF-8 (e.g. an image-input chat completion that encodes binary), `from_utf8_lossy` silently replaces bytes with U+FFFD. This is fine for v0.1.0 (chat-completion bodies are JSON) but blocks future byte-exact replay.
- **Fix sketch:** Detect non-UTF-8 once per body; switch to a `base64(...)` string under a sibling field `input_base64` / `output_base64` and leave the lossy `input` / `output` as a human-readable fallback. Bump schema version.
- **Verification:** Round-trip a non-UTF-8 body and confirm the base64 field reconstructs byte-exact.

## From the manual-tester exploratory pass (2026-05-26)

### BUG-1 — Low — 405 Method Not Allowed returns an empty body, inconsistent with the JSON 404

- **Source:** Manual-tester report, scenario #21.
- **Where:** Axum router fallback path for `POST /v1/chat/completions` (and other allowed-method-only routes). The JSON 404 is implemented in `crates/replayable-proxy/src/proxy.rs::not_found` (or equivalent); the 405 emerges from axum's default `MethodNotAllowed` response and has no body.
- **Repro:** `curl -X GET http://127.0.0.1:8080/v1/chat/completions` — returns empty body with only `Allow: POST` header.
- **Impact:** Low — RFC-compliant, but inconsistent with the proxy's own JSON-error convention. Operator UX nit; tooling that parses the JSON envelope on 4xx will fail on 405.
- **Fix sketch:** Override the `MethodNotAllowed` handler in the router (or layer a `tower` `MapResponseLayer`) to emit the same `{"error": "..."}` JSON shape used by 404.
- **Verification:** Add `unknown_method_returns_stable_json_error_shape` to `tests/integration.rs` mirroring `unknown_path_returns_stable_json_error_shape`.

### BUG-2 — Low — No startup WARN when `REPLAYABLE_LISTEN` is non-loopback

- **Source:** Manual-tester report, scenario #23. Cross-references the original H3 fix recommendation #3 (review section on H3) which suggested a startup `warn!` line when `listen.ip()` is `0.0.0.0`.
- **Where:** `crates/replayable-proxy/src/main.rs` — startup logging around the `listening` INFO line.
- **Problem:** When an operator explicitly sets `REPLAYABLE_LISTEN=0.0.0.0:8080`, the proxy boots and binds silently. The defense-in-depth WARN recommended during the H3 review was not implemented because the H4 fix made loopback the default and the team judged the visibility gain marginal. Manual testing surfaced it as a genuine UX gap for operators who DO opt in to LAN exposure.
- **Impact:** Low — secure default already mitigates the risk. WARN is purely a "you did this on purpose, right?" signal in the operator's log aggregator.
- **Fix sketch:**
  ```rust
  if !config.listen.ip().is_loopback() {
      warn!(
          listen = %config.listen,
          "NON-LOOPBACK LISTEN — the proxy will accept connections from outside this host. Verify your network ACLs.",
      );
  }
  ```
- **Verification:** Add an integration test that boots the proxy with `0.0.0.0:0` and asserts the WARN line appears in captured stderr.
